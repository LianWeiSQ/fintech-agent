# Fitech Agent 优化计划

## 目标

- 统一项目名、包名、CLI、配置项与产物路径，全部对齐到 `fitech-agent` / `fitech_agent`。
- 在不改变 V1 产品边界的前提下，提升多 Agent 新闻流水线的可维护性与可扩展性。
- 为后续接入更多数据源、模型路由与自动化交付打好工程基础。

## 当前观察

- 主流程已经清晰，但采集、分析、存储、报告仍集中在 `NewsPipeline` 中，耦合偏高。
- `.env`、TOML 配置、模型路由已经可用，但命名、配置入口与密钥管理分散在多个位置。
- SQLite 审计链路可满足回放需求，但还缺少 schema 版本管理、索引设计与数据留存策略。
- 现有测试覆盖了规范化、审计和端到端主流程，但对 CLI、配置优先级和打包回归覆盖还不够。

## 优化路线

### P0 - 命名与打包一致性

1. 保持发行名、控制台命令、Python 包、LangGraph 入口、环境变量和数据库路径都统一为 `fitech-agent`。
2. 将 `.egg-info` 视为发布生成物，发版时自动重建，避免手工维护导致元数据漂移。
3. 增加一份轻量发布检查清单，至少覆盖安装、CLI 调用、示例跑通和报告产物校验。

### P1 - 架构解耦

1. 将 `NewsPipeline` 拆分为运行生命周期、新闻采集、事件分析、报告发布等更清晰的服务层。
2. 引入显式的流水线状态模型，减少节点之间直接传递大字典带来的隐式约束。
3. 将 prompt、来源可信度规则、资产映射规则从流程代码中抽离到独立模块或配置资产。

### P1 - 数据与存储加固

1. 为 SQLite 增加 schema 版本与迁移脚本，避免后续表结构调整只能手工清库。
2. 为高频查询增加索引，例如 `stage_payloads(run_id, entity_type)` 与 `outcomes(run_id, assessment_id)`。
3. 明确原始 payload、报告文件、评估结果的留存与导出策略，控制审计数据体积。

### P2 - 可靠性与可观测性

1. 为数据源失败、降级运行、LLM fallback、报告渲染告警补充结构化日志。
2. 记录各阶段耗时、数据源采集量和失败率，方便定位慢点与脆弱环节。
3. 在任务开始前增加 source、timezone、model route 等配置校验，尽量提前失败。

### P2 - 质量与交付

1. 扩展测试到 CLI、`.env` / TOML 优先级、LangGraph 启动和异常恢复场景。
2. 增加 lint、format、type check、pytest 的 CI，确保后续重构可持续演进。
3. 补充开发者文档，降低新增数据源适配器、模型提供方与报告模块的改造成本。

## 建议的下一轮迭代

1. 先锁定本次命名重构，为导入路径、CLI 命令和环境变量补齐回归测试。
2. 再抽离 pipeline state 与 storage 访问边界，降低主流程复杂度。
3. 最后补上最小 CI 与打包校验，再继续接入更多实时数据源。
